# Skill: self-check-code

## Назначение

Этот скилл нужен для задач, где надо **писать, править, рефакторить, проверять и добивать код до рабочего состояния**.
Главное правило: **сначала сделать, потом проверить, потом отвечать**.

Скилл обязателен для:
- создания нового кода;
- правок в существующем проекте;
- фиксов багов;
- рефакторинга;
- интеграций API;
- изменений схем, маршрутов, меню, форм, обработчиков, ботов, UI и серверной логики.

## Цель

Давать не "похоже работает", а результат с самопроверкой:
- код собран;
- синтаксис живой;
- импорты не сломаны;
- затронутые сценарии проверены;
- ответ краткий и предметный.

## Принципы

1. Не писать код вслепую. Сначала прочитать структуру проекта.
2. Не плодить лишние файлы и абстракции.
3. Менять минимально необходимую поверхность.
4. После изменений запускать проверки, а не гадать.
5. Если проверка упала — сначала чинить, потом снова гонять.
6. Не врать о проверках. Если не запускал — так и писать.
7. Не слать пользователю простыню. Только: что сделал, что проверил, где риск.

## Обязательный порядок работы

### 1. Разобрать проект
Перед правкой определить:
- стек;
- entrypoint;
- команды запуска;
- тесты;
- сборку;
- типизацию;
- линтер;
- где живут схемы, роуты, обработчики, UI, env.

Минимум проверить:
- `package.json`
- `pyproject.toml` / `requirements.txt`
- `Makefile`
- `README*`
- корневые папки `src`, `app`, `server`, `client`, `bot`, `api`, `shared`, `tests`

### 2. Спланировать изменение
До правки ответить себе на 4 вопроса:
- что именно меняем;
- какие файлы затронем;
- что может сломаться рядом;
- чем это проверить.

### 3. Внести правки
Править только то, что нужно для результата.

Обязательно проверить связки:
- импорт -> экспорт;
- тип -> использование;
- schema -> route -> handler -> UI;
- callback_data -> handler;
- env -> config -> runtime;
- request -> formatter -> output.

### 4. Самопроверка после правки
Скилл **не завершает задачу**, пока не выполнит подходящие проверки.

Порядок проверок:
1. Синтаксис
2. Типы
3. Сборка
4. Тесты
5. Точечный smoke по затронутому сценарию

Если полного набора нет — брать доступный минимум.

## Матрица проверок

### JS / TS
Проверять в таком порядке, если команды доступны:
- `npm run lint`
- `npm run typecheck`
- `npm run build`
- `npm test`

Если отдельных команд нет:
- искать `pnpm`, `yarn`, `bun`;
- брать команды из `package.json`;
- если тестов нет, минимум — `build` или `tsc`.

### Python
Проверять в таком порядке, если доступно:
- `python -m compileall .`
- `pytest`
- `ruff check .`
- `mypy .`

Если тестов нет, минимум — компиляция и точечный запуск затронутого модуля.

### Node bot / Telegram bot
Дополнительно проверять:
- разбор входящего текста;
- меню;
- reply/inline кнопки;
- callback handlers;
- падение на пустых данных;
- текст ответа при `null` / `[]` / ошибке API.

### API / backend
Дополнительно проверять:
- валидацию входа;
- коды ответа;
- null safety;
- таймауты внешних запросов;
- формат ошибок;
- обратную совместимость контрактов.

### Frontend / dashboard
Дополнительно проверять:
- форма сохраняет данные;
- обязательные поля помечены;
- пустые значения не роняют UI;
- кнопки отключаются в loading/error;
- данные после reload читаются корректно.

## Правила самокоррекции

Если проверка упала:
1. зафиксировать причину;
2. исправить;
3. прогнать повторно;
4. только после этого идти дальше.

Не останавливаться на первом фейле, если его можно исправить локально.

## Стоп-условия

Можно завершать задачу только если выполнено одно из двух:

### Вариант А — всё прошло
Есть минимум одна реальная проверка на изменённый сценарий.

### Вариант Б — есть честный блокер
Например:
- нет токена;
- закрыт внешний API;
- нет зависимостей;
- проект не ставится;
- отсутствует база.

Тогда в ответе нужен только конкретный блокер, без воды.

## Формат финального ответа

Ответ всегда короткий.

Шаблон:

```text
Сделано:
- ...

Проверил:
- ...

Риск:
- ...
```

Если рисков нет:

```text
Риск:
- не вижу
```

Если проверка не запускалась, запрещено писать "готово".
Нужно писать так:

```text
Проверил:
- синтаксис не гонял
- сборку не гонял
- причина: нет зависимостей / нет доступа / нет ключей
```

## Антипаттерны

Запрещено:
- писать "должно работать" без проверки;
- показывать сырые JSON-простыни вместо результата;
- расписывать теорию вместо фикса;
- менять полпроекта ради локальной задачи;
- прятать непроверенные места;
- смешивать разные сущности и сценарии в одном formatter/handler, если от этого ломается логика.

## Чеклист перед ответом

- прочитал структуру проекта;
- понял точку входа;
- изменил только нужные файлы;
- проверил импорты и вызовы;
- прогнал доступные проверки;
- перепроверил затронутый сценарий;
- в ответе нет воды.
